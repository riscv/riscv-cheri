== The {rv32y_uni_base_name} Capability Base for RV32

This section describes an in-memory format and properties of a capability encoding intended for RV32.
This format is based upon the <<rv64y_cap_description,{rv64y_uni_base_name}>> format with the following changes:

* The width, or presence, of fields in the encoding are changed as defined in <<rvy32_uni_base_name_param_summary>>.
* The architectural permissions (AP) field is compressed to save encoding space, and so rules are defined for removing permissions.

[#AP-field-encoding32, reftext="Compressed AP field"]
==== Architectural Permissions (AP) Encoding - pc1

The <<AP-field>> is {cap_rv32_perms_width} bits wide and is encoded using a compressed representation avoiding unused combinations to save encoding space.
The <<AP-field>> is split into four quadrants.
The quadrant number is taken from AP[4:3].
The meanings of bits AP[2:0] for each quadrant are shown in <<cap_perms_encoding_nolevels32>>.

NOTE: Extensions which allocate new permissions need to specify modifications to the AP-fields and the <<CLRPERM>> rules to reduce the permissions to maintain an encodable set.
 The recommendation is for new permissions added to existing entries in quadrant 0 _must not_ be granted as that quadrant includes the <<null-cap>> capability.
 New permissions added to quadrant 1 entry 0 _must_ be granted as this represents both of the <<root-rx-cap>> and <<root-rw-cap>> capabilities.

The compressed permission format specifies a procedure for encoding architectural permissions.
When encoding permissions the following rules are run once _in order_:

[#clrperm-rules]
.Compressed <<AP-field>> <<CLRPERM>> rules.
[float="center",align="center",cols="1,1,4",options="header"]
|===
| <<CLRPERM>> Rule | Permission   | Valid only if
| `RV32-base-1`    | C            | R (supersedes <<perm_req:base:c:r-or-w>>)
| `RV32-base-2`    | X            | R
| `RV32-base-3`    | W            | not\(C) or LM
| `RV32-base-4`    | X            | W or C
| `RV32-base-5`    | LM           | C (supersedes <<perm_req:base:lm:c-and-r>>)
| `RV32-base-6`    | X            | (C and LM) or not (C or LM)
| `RV32-base-7`    | ASR          | W and C and X (supersedes <<perm_req:base:asr:x>>)
|===

=== {cheri_levels1_ext_name} support

{cheri_levels1_ext_name} adds <<zylevels1_lg_perm>> and <<zylevels1_sl_perm>> to the states in <<cap_perms_encoding_nolevels32>>, including allocating some previously reserved entries.
<<zylevels1_alloc_into_ap_field>> and <<zylevels1_mods_into_ap_field>> show which entries have been allocated or modified to grant LG and/or SL.

[#zylevels1_alloc_into_ap_field]

| 2        | 7       | 0      | 0  | R, W, C, LM granted
| 3        | 6       | 1      | 1  | R, W, C, LM, LG, SL granted
|===
