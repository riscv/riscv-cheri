== "{cheri_levels1_ext_name}" Extension for CHERI 2-Level Information Flow Control

{cheri_levels1_ext_name} adds an *Information Flow Control* (IFC) mechanism to CHERI.
It adds one new "level" metadata bit to classify capabilities as either *global* or *local*, adds checks to memory operations to control how capabilities can be shared, and adds two new level-related permissions, **Store Local (SL)** and **Load Global (LG)**.

=== Added Metadata Bits (Permissions and Flags)

[#zylevels1_ap_field_summary,width="100%",options=header,halign=center,cols="2,2,5"]
|==============================================================================
| Bit                    | Type                   | Comment
| <<zylevels1_gl_perm>>  | Information flow label | Used to classify capabilities as global or local.
| <<zylevels1_sl_perm>>  | Data memory permission | Used to filter the {ctag}s of stored capabilities.
| <<zylevels1_lg_perm>>  | Data memory permission | Used to filter the permissions of loaded capabilities.
|==============================================================================

ifdef::cheri_v9_annotations[]
NOTE: The <<zylevels1_gl_perm>> and <<zylevels1_sl_perm>> are as in CHERI v9, Morello, and CHERIoT v1.
The <<zylevels1_lg_perm>> is as in CHERIoT v1.
endif::[]

[#zylevels1_gl_perm,reftext="GL(obal) Flag"]
==== The Capability Global (GL) Flag

The _Capability Global_ (GL) flag is a metadata bit used to track and control the propagation of capabilities.
It works in conjunction with the <<zylevels1_lg_perm>> and <<zylevels1_sl_perm>> permissions to enforce information flow boundaries.

The _Capability Global_ flag holds one of two values:

* 1: the capability is _global_.
* 0: the capability is _local_.

Like permissions, the _Capability Global_ flag can be cleared when creating a new capability from an existing one, but it can never be set unless derived from a superset capability which has the  _Capability Global_ flag set.

[#zylevels1_sl_perm,reftext="SL-permission"]
==== Store Local Permission (SL)

Store Local Permission (SL) controls whether local capabilities can be stored to memory.
Storing a local capability (one without the <<zylevels1_gl_perm>> set) using an authorizing capability that lacks <<zylevels1_sl_perm>> results in the stored capability having its {ctag} set to zero.

<<zylevels1_sl_perm>> is dependent on both <<c_perm>> and <<w_perm>>.
If either of these permissions is clear, <<zylevels1_sl_perm>> is ignored and does not need to be encodable in the capability format.

[#zylevels1_lg_perm,reftext="LG-permission"]
==== Load Global Permission (LG)

Load Global Permission (LG) controls whether global capabilities can be loaded from memory.
Loading a capability using an authorizing capability that lacks <<zylevels1_lg_perm>> clears the <<zylevels1_gl_perm>> bit of the loaded capability, making it local.
If the loaded capability is _unsealed_, its <<zylevels1_lg_perm>> is also cleared.

This behavior is similar to how <<lm_perm>> affects loaded capabilities (but note the difference in interaction with seals).

<<zylevels1_lg_perm>> is dependent on both <<c_perm>> and <<r_perm>>.
If either of these permissions is clear, <<zylevels1_lg_perm>> is ignored and does not need to be encodable in the capability format.

=== Interaction with <<root-cap>> Capabilities

The <<root-cap>> capabilities used in the system are extended:

* The definitions of <<root-rx-cap>> and <<root-rw-cap>> capabilities are both augmented to require that <<zylevels1_gl_perm>> be set to _global_.

* A <<root-rx-cap>> capability is required to grant <<zylevels1_lg_perm>>.

* A <<root-rw-cap>> capability is required to grant both <<zylevels1_lg_perm>> and <<zylevels1_sl_perm>>.

==== <<CLRPERM>> and the Capability Global (GL) Flag
<<CLRPERM>> can produce a new capability with its <<zylevels1_gl_perm>> cleared, even if the source capability is sealed.
This differs from architectural and software permissions.
This applies to both the implicit clearing of permissions during memory loads and explicit <<CLRPERM>> instructions.

[#zylevels1-clrperm-rules]
==== Additional <<CLRPERM>> rules

As mentioned, the <<zylevels1_sl_perm>> and <<zylevels1_lg_perm>> permissions are dependent on base permissions.
<<CLRPERM>> operations (including implicit ones) and/or the capability encoding clear these permissions when their dependencies are cleared.
Specifically, we add the following rules to those of <<sec_permission_transitions>>:

| [[perm_req:zylevels1:sl:c-and-w,{cheri_levels1_ext_name}-2]]{cheri_levels1_ext_name}-2 +

=== Interaction with <<LOAD_CAP>>

As outlined above, {cheri_levels1_ext_name} introduces two new constraints on capabilities loaded from memory.
These constraints apply to <<LOAD_CAP>> (`{load_cap_name_lc} {cd}, offset({cs1})`) and other instructions with similar semantics.
The behavior can be described as implicitly reducing the permissions of the loaded capability:

* If the authority (`{cs1}`) lacks <<zylevels1_lg_perm>> and the loaded capability is unsealed, both its <<zylevels1_gl_perm>> and <<zylevels1_lg_perm>> are cleared in the result register (`{cd}`).

* If the authority (`{cs1}`) lacks <<zylevels1_lg_perm>> and the loaded capability is sealed, its <<zylevels1_gl_perm>> is cleared in the result register (`{cd}`).
+
NOTE: The absence of <<zylevels1_lg_perm>> affects the <<zylevels1_gl_perm>> even of sealed capabilities because the latter is an information flow label rather than a architectural permission.

.<<zylevels1_lg_perm>> effects for loading capabilities

NOTE: <<zylevels1_lg_perm>> is only effective when both <<r_perm>> and <<c_perm>> are granted.

NOTE: Because <<zylevels1_sl_perm>> is only relevant to capabilities granting <<w_perm>>, any load that clears <<w_perm>> (e.g., due to a lack of <<lm_perm>> permission) will also necessarily clear <<zylevels1_sl_perm>>.

=== Interaction with <<STORE_CAP>>

{cheri_levels1_ext_name} adds a constraint to <<STORE_CAP>> (`{store_cap_name_lc} {cs2}, offset({cs1})`) and similar instructions.
The capability is stored with a {ctag} set to one only if:

* The capability being stored (`{cs2}`) is global (<<zylevels1_gl_perm>>=1), OR
* The authority used for the store (`{cs1}`) grants <<zylevels1_sl_perm>>.

Otherwise, the capability is stored with the {ctag} set to zero.

NOTE: While a lack of<<zylevels1_lg_perm>> restricts flow by reducing permissions (labeling the loaded capability as local and stripping permissions), <<zylevels1_sl_perm>> restricts flow by invalidating capabilities (by setting the {ctag} to zero).

=== Interaction with <<SCSS>>

In implementations of {cheri_levels1_ext_name}, the <<SCSS>> instruction (`{SCSS_LC} rd, {cs1}, {cs2}`) indicates that `{cs2}` is a subset of `{cs1}` (setting `rd` to 1) only if:

* The Global flag of the supposed subset capability (`{cs2}`) is clear (0), OR
* The Global flag of the supposed superset capability (`{cs1}`) is set (1).

The existing permission subset logic applies to the new <<zylevels1_sl_perm>> and <<zylevels1_lg_perm>> permissions.

=== Interaction with <<CBLD>>

A <<CBLD>> instruction (`{CBLD_LC} {cd}, {cs1}, {cs2}`) may yield a valid capability (`{cd}.tag=1`) only if:

* The Global flag of the bits to be reconstituted (`{cs2}`) is clear (0), OR
* The Global flag of the reconstituting superset capability (`{cs1}`) is set (1).

The existing permission subset logic applies to the new <<zylevels1_sl_perm>> and <<zylevels1_lg_perm>> permissions.

=== Interaction with <<YSUNSEAL>>

The <<YSUNSEAL>> instruction (`{YSUNSEAL_LC} {cd}, {cs1}, {cs2}`) ensures that the result has <<zylevels1_gl_perm>> set only if the unsealing authority (`{cs1}`) also has <<zylevels1_gl_perm>> set.
(That is, `{cd}` is global if and only if both the authority in `{cs1}` and the unsealed form of `{cs2}` are global.)

The existing permission subset logic applies to the new <<zylevels1_sl_perm>> and <<zylevels1_lg_perm>> permissions.

=== Rationale and Software Use Cases

The primary goal of {cheri_levels1_ext_name} is to prevent storing pointers to specific memory regions in order to enforce security invariants.
There are many uses of this mechanism, but two examples are listed below.

==== Restricting Capability Flow Between Regions or Domains (e.g., Stack-to-Heap or Kernel-to-User)

Software can use these features to prevent pointers from leaking between different memory regions or privilege domains that have different security properties.
The general pattern is to mark sensitive capabilities as local (by clearing their <<zylevels1_gl_perm>>) and ensuring that destination memory regions are accessed through authorities that lack the <<zylevels1_sl_perm>>.

*   **Intra-program flow (e.g., Stack-to-Heap):**
    To prevent pointers to short-lived memory (like the stack) from leaking into long-lived memory (like the heap), software marks all stack pointers as local.
    If the authorities for the heap lack the <<zylevels1_sl_perm>>, any attempt to store a stack pointer into the heap results in the hardware invalidating the stored capability.
    This ensures stack pointers are strictly confined to registers and the stack itself.

*   **Cross-domain flow (e.g., Kernel-to-User):**
    These features also apply across privilege boundaries.
    For example, a kernel can ensure that the capability authorizing a write to userspace (e.g., for a `copy_to_user` operation) lacks the <<zylevels1_sl_perm>>.
    Even if a kernel bug causes it to accidentally attempt to copy sensitive, local kernel capabilities into the user's buffer, the hardware will invalidate those capabilities.
    This prevents unintentional leaks of usable capabilities to a less-privileged domain.

==== Enforcing Shallow vs. Deep No-Capture Guarantees

When passing a capability to a less-trusted function, a caller often wants to enforce "no-capture" guarantees, ensuring the callee cannot hold onto the capability after returning.
{cheri_levels1_ext_name} provides two levels of protection against capability capture.

*   **Shallow no-capture (via <<zylevels1_gl_perm>> and <<zylevels1_sl_perm>>):**
    The caller makes the argument capability local by clearing its <<zylevels1_gl_perm>>.
    Because the callee's global variables and persistent data structures are accessed through capabilities that lack the <<zylevels1_sl_perm>>, the callee cannot store the local argument capability into them.
    This enforces a shallow guarantee: the callee cannot capture the argument itself, but it might still load a global capability _through_ the argument and capture that.

*   **Deep no-capture (via <<zylevels1_lg_perm>>):**
    To solve the shallow limitation, the caller also removes the <<zylevels1_lg_perm>> from the argument capability.
    If the callee loads any capability through this argument, the hardware automatically removes the loaded capability's <<zylevels1_gl_perm>> and <<zylevels1_lg_perm>>.
    This enforces a deep no-capture guarantee: the callee is prevented from capturing the argument _and_ anything reachable from it.
