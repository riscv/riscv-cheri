== "{cheri_levels1_ext_name}" Extension for CHERI 2-Level Information Flow Control

{cheri_levels1_ext_name} adds an *Information Flow Control* (IFC) mechanism to CHERI.
It adds one new "level" metadata bit (<<zylevels1_gl_flag>>) to classify capabilities as either *global* or *local*, adds checks to memory operations to control how capabilities can be shared, and adds two new level-related permissions, <<zylevels1_sl_perm,Store Local (SL)>> and <<zylevels1_lg_perm,Load Global (LG)>>.

NOTE: The behavior is an extension as it may be replaced by a future scheme with more than two levels.

=== Added Metadata Bits (Permissions and Flags)

[#zylevels1_ap_field_summary,width="100%",options=header,halign=center,cols="2,2,5"]
|==============================================================================
| Bit                    | Type                   | Comment
| <<zylevels1_gl_flag>>  | Information flow label | Used to classify capabilities as global or local.
| <<zylevels1_sl_perm>>  | Data memory permission | Used to filter the {ctag}s of stored capabilities.
| <<zylevels1_lg_perm>>  | Data memory permission | Used to filter the permissions of loaded capabilities.
|==============================================================================

[#zylevels1_gl_flag,reftext="GL(obal) Flag"]
==== The Capability Global (GL) Flag

The <<zylevels1_gl_flag>> is a metadata bit used to track and control the propagation of capabilities.
It works in conjunction with the <<zylevels1_lg_perm>> and <<zylevels1_sl_perm>> permissions to enforce information flow boundaries.

[#levels_globalcap,reftext="global capability"]
Global Capability::
A global capability has <<zylevels1_gl_flag>>=1.

[#levels_localcap,reftext="local capability"]
Local Capability::
A local capability has <<zylevels1_gl_flag>>=0.

Like permissions, the <<zylevels1_gl_flag>> can be cleared when creating a new capability from an existing one, but it can never be set unless derived from a superset capability which has the <<zylevels1_gl_flag>> set.

[#zylevels1_sl_perm,reftext="SL-permission"]
==== Store Local Permission (SL)

Store Local Permission (SL) controls whether a <<levels_localcap>> can be stored to memory.

If the authorizing capability lacks <<zylevels1_sl_perm>> then the stored capability has its {ctag} set to zero.

<<zylevels1_sl_perm>> is dependent on both <<c_perm>> and <<w_perm>> as shown in <<zylevels1_clrperm_table>>.

[#zylevels1_lg_perm,reftext="LG-permission"]
==== Load Global Permission (LG)

Load Global Permission (LG) controls whether <<levels_globalcap, global capabilities>> can be loaded from memory.
Loading a capability using an authorizing capability that lacks <<zylevels1_lg_perm>> clears the <<zylevels1_gl_flag>> bit of the loaded capability, making it into a <<levels_localcap>>.
If the loaded capability is unsealed, its <<zylevels1_lg_perm>> is also cleared.

NOTE: This behavior is similar to how <<lm_perm>> affects loaded capabilities (but note the difference in interaction with seals).

<<zylevels1_lg_perm>> is dependent on both <<c_perm>> and <<r_perm>> as shown in <<zylevels1_clrperm_table>>.

=== Interaction with <<root-cap>> Capabilities

The <<root-cap>> capabilities definitions are extended:

* A <<root-rx-cap>> capability is required to grant <<zylevels1_lg_perm>> and to be a <<levels_globalcap>>.

* A <<root-rw-cap>> capability is required to grant both <<zylevels1_lg_perm>> and <<zylevels1_sl_perm>> and to be a <<levels_globalcap>>.

==== <<CLRPERM>> and the Capability Global (GL) Flag
<<CLRPERM>> can produce a new capability with its <<zylevels1_gl_flag>> cleared, even if the source capability is sealed.
This differs from architectural and software permissions.
This applies to both the implicit clearing of permissions during memory loads and explicit <<CLRPERM>> instructions.

=== Interaction with <<LOAD_CAP>>

{cheri_levels1_ext_name} introduces two new constraints on capabilities loaded from memory.
These constraints apply to <<LOAD_CAP>> and all other instructions that can load valid capabilities.

The permissions of the loaded capability may be reduced:

* If the authorizing capability lacks <<zylevels1_lg_perm>> and the loaded capability is unsealed, both <<zylevels1_gl_flag>> and <<zylevels1_lg_perm>> are cleared in the result.

* If the authorizing capability lacks <<zylevels1_lg_perm>> and the loaded capability is sealed, <<zylevels1_gl_flag>> is cleared in the result.
+
NOTE: The absence of <<zylevels1_lg_perm>> affects the <<zylevels1_gl_flag>> even of sealed capabilities because the latter is an information flow label rather than an architectural permission.

=== Interaction with <<STORE_CAP>>

{cheri_levels1_ext_name} adds a constraint to <<STORE_CAP>> and all instructions which can store valid capabilities.

The capability is stored with a {ctag} set to one only if:

* The capability being stored is a <<levels_globalcap>>, or
* The authorizing capability grants <<zylevels1_sl_perm>>.

Otherwise, the capability is stored with the {ctag} set to zero.

NOTE: While a lack of <<zylevels1_lg_perm>> restricts flow by reducing permissions (labeling the loaded capability as local and stripping permissions), <<zylevels1_sl_perm>> restricts flow by invalidating capabilities (by setting the {ctag} to zero).

=== Interaction with Capability Subsets

The definition of <<cap_subset>> has an additional rule:

* The <<zylevels1_gl_flag>> of `{cs2}` is zero, or the <<zylevels1_gl_flag>> of `{cs1}` is one.

NOTE: This affects the behavior of <<SCSS>>, <<CBLD>> and <<YSUNSEAL>>.

=== Rationale and Software Use Cases

The primary goal of {cheri_levels1_ext_name} is to prevent storing pointers to specific memory regions in order to enforce security invariants.
There are many uses of this mechanism, but two examples are listed below.

==== Restricting Capability Flow Between Regions or Domains (e.g., Stack-to-Heap or Kernel-to-User)

Software can use these features to prevent pointers from leaking between different memory regions or privilege domains that have different security properties.
The general pattern is to mark sensitive capabilities as local (by clearing their <<zylevels1_gl_flag>>) and ensuring that destination memory regions are accessed through authorities that lack the <<zylevels1_sl_perm>>.

*   **Intra-program flow (e.g., Stack-to-Heap):**
    To prevent pointers to short-lived memory (like the stack) from leaking into long-lived memory (like the heap), software marks all stack pointers as local.
    If the authorities for the heap lack the <<zylevels1_sl_perm>>, any attempt to store a stack pointer into the heap results in the hardware invalidating the stored capability.
    This ensures stack pointers are strictly confined to registers and the stack itself.

*   **Cross-domain flow (e.g., Kernel-to-User):**
    These features also apply across privilege boundaries.
    For example, a kernel can ensure that the capability authorizing a write to userspace (e.g., for a `copy_to_user` operation) lacks the <<zylevels1_sl_perm>>.
    Even if a kernel bug causes it to accidentally attempt to copy sensitive, local kernel capabilities into the user's buffer, the hardware will invalidate those capabilities.
    This prevents unintentional leaks of usable capabilities to a less-privileged domain.

==== Enforcing Shallow vs. Deep No-Capture Guarantees

When passing a capability to a less-trusted function, a caller often wants to enforce "no-capture" guarantees, ensuring the callee cannot hold onto the capability after returning.
{cheri_levels1_ext_name} provides two levels of protection against capability capture.

*   **Shallow no-capture (via <<zylevels1_gl_flag>> and <<zylevels1_sl_perm>>):**
    The caller makes the argument capability local by clearing its <<zylevels1_gl_flag>>.
    Because the callee's global variables and persistent data structures are accessed through capabilities that lack the <<zylevels1_sl_perm>>, the callee cannot store the local argument capability into them.
    This enforces a shallow guarantee: the callee cannot capture the argument itself, but it might still load a <<levels_globalcap>> _through_ the argument and capture that.

*   **Deep no-capture (via <<zylevels1_lg_perm>>):**
    To solve the shallow limitation, the caller also removes the <<zylevels1_lg_perm>> from the argument capability.
    If the callee loads any capability through this argument, the hardware automatically removes the loaded capability's <<zylevels1_gl_flag>> and <<zylevels1_lg_perm>>.
    This enforces a deep no-capture guarantee: the callee is prevented from capturing the argument _and_ anything reachable from it.
